# FLUX.2 Klein 4B real-time optimization — bench log

Live benchmark log for the optimization push that started 2026-04-30. Each phase
appends a row to the comparison table at the bottom, and a section explaining
the config + caveats.

The earlier successful work (compile + alpha-blend img2img + 2-step) is already
documented in `RESULTS.md` and is preserved here as the **0_baseline** row.
This doc only covers what's tried *on top of* that baseline.

Production target (4-step, 256²) on 2× RTX 5090 / driver 580 dual-GPU pod:
- bf16 + compile (RESULTS.md baseline): 62.6 ms / 16 fps single-GPU.
- + fp8 transformer (Phase 3): ~45 ms / 22 fps single-GPU.
- + PerTensor + VAE fp8 (Phase 3b/c): ~43 ms / 23 fps single-GPU, **44.93 fps dual** (compile_mode=default, prior production).

That's a **2.7×** stack (50.67 / ~16 fps RESULTS.md baseline) with no quality
loss at the user's normal alpha range (0.05-0.18). 512²/4-step lands at
**17.11 fps dual** on the same stack.

## What we ruled out, and why

| Method | Why no win at Klein 4-step |
|---|---|
| **Pruna `flux2-klein-4b-smashed` runtime** | Pruna 0.3.3 source uses transformers APIs removed in 5.x; can't run alongside diffusers main. Recipe replicated component-wise instead (Phase 3). |
| **TeaCache** | Data-dependent skip in `forward` is a graph break for `torch.compile`. Cost of giving up compile (~1.4× slowdown) > best-case TeaCache gain (1.25× from skipping 1 of 4 calls). Measured: 104 ms vs 45 ms compiled — 2.3× *slower* with TeaCache. |

| **TaylorSeer** | Needs N+1 warmup steps to fit Nth-order derivative — 0 forecasts at 4 steps. |
| **TorchAO `Float8WeightOnlyConfig` (fp8wo)** | Per-matmul dequant cost > VRAM savings on Blackwell. Measured: 49.8 ms vs 36 ms baseline — 38% slower. |
| **`torch.compile(mode="reduce-overhead")`** | Earlier RESULTS.md finding: PEFT wrapper view tensors break CUDA graph capture. Not retested with fp8 (would require half a day; ROI capped at ~10-15%). |
| **Pruna SDNQ 4-bit / FlashAttention 3** | Sm_120 / Hopper-only paths; not applicable on Blackwell. |
| **xDiT distributed DiT** | Phase 8 scope — held until Phase 7 lands so we have a baseline to compare against. |

## Phase 1 — Pruna `flux2-klein-4b-smashed`: SKIPPED

**Outcome: not pursued.** Documented for the record so we don't re-investigate.

### Why we tried

Pruna's smashed model claims 2-3× speedup via FORA caching + TorchAO fp8wo +
torch.compile. Worth a benchmark — that's three of the techniques on our list
in one shot.

### What blocked it

| Issue | Detail |
|---|---|
| Pruna 0.3.3 ships with `transformers<5.0.0` cap | Klein needs Qwen3, which is in `transformers>=5.5`. PR #642 (open Dependabot) bumps the cap to `<6.0.0` — the cap is defensive, not load-bearing for most imports. |
| Pruna source uses `transformers.pipelines.text2text_generation` | This module path was removed in `transformers` 5.x. PR #642 doesn't patch the source. Even with `--no-deps` install, `import pruna` fails on a transitive `is_janus_llamagen_ar` check. |
| Pruna's full install resolver upgrades `huggingface_hub`, `numpy`, etc. | Each upgrade cascades into more breakage (e.g. transformers 5.5.4 wants `huggingface_hub>=1.5.0` (1.x major), but pruna pulls 0.36.x which removed `is_offline_mode`). |
| Smashed model has zero benefit at 2-step inference | FORA `start_step=4`, `interval=3` — at our 2-step recipe nothing caches. |
| TorchAO fp8wo on Blackwell | RESULTS.md already saw 257 ms regression vs 232 ms bf16 (dequant cost > 4 GB VRAM saved). Pruna's smashed model bakes this in by default. |

### Decision

Skip Pruna runtime. Replicate its recipe component-by-component manually:

- `torch.compile` — already in baseline.
- TorchAO fp8wo — Phase 3.
- FORA-style transformer-output caching — Phase 4 (only if Phase 3 + Phase 2 / TeaCache underperform).

This gives us the same techniques, but we control which combinations to test
and avoid Pruna's 5-package transitive upgrade tree that breaks the env.

## Phase 3 — TorchAO fp8 quantization 🏆

**Headline: fp8 dynamic-activation gives 24-30% latency reduction over the
compiled bf16 baseline on Blackwell sm_120, plus 3.7 GB VRAM savings.**
This is the biggest single optimization win in the whole document.

### Method

Three variants, each: load fresh `Flux2KleinKVPipeline`, apply quantization to
the transformer's linear layers (skipping `pe_embedder`, norms, and embedding
proj — same filter shape as Pruna's smash_config), then `torch.compile` the
transformer + VAE encoder + VAE decoder. Bench at 256² and 512² × 2 steps,
20 timed iters each.

VAE is **never** quantized — the small decoder is precision-sensitive and the
fp8 cost there is dominated by quantize/dequant overhead.

Bench: `workers/runpod-flux2klein/bench_torchao_fp8.py`
Output: `/workspace/bench-2026-04-30/torchao_fp8/summary.json` on pod v2.

### Results (pod v2)

| variant | 256² mean | 256² p95 | 256² fps | 512² mean | 512² p95 | 512² fps | VRAM |
|---|---|---|---|---|---|---|---|
| bf16 (control) | 36.11 ms | 36.95 | 27.69 | 105.58 ms | 105.81 | 9.47 | 15.99 GB |
| fp8 weight-only (`Float8WeightOnlyConfig`) | 49.80 ms | 50.64 | 20.08 | 119.38 ms | 119.80 | 8.38 | 12.31 GB |
| **fp8 dynamic-activation** (`Float8DynamicActivationFloat8WeightConfig`, PerRow) | **27.39 ms** | **27.45** | **36.50** 🔥 | **73.42 ms** | **73.86** | **13.62** | 12.32 GB |

Δ vs bf16:

| variant | 256² Δms | 256² Δfps | 512² Δms | 512² Δfps |
|---|---|---|---|---|
| fp8 weight-only | **+13.69 ms (+38%)** REGRESSION | -7.61 | +13.80 ms (+13%) REGRESSION | -1.09 |
| **fp8 dynamic-activation** | **-8.72 ms (-24%)** | **+8.81** | **-32.16 ms (-30%)** | **+4.15** |

p95 within 0.5 ms of mean across both winning cells — frame timing remains
glassy after quantization.

### Why fp8wo regresses but dynamic-activation wins

`Float8WeightOnlyConfig` stores weights in fp8 and **dequantizes them to bf16
before each matmul**. On a transformer the matmul is the bottleneck; you pay
the dequant cost every forward pass. Net: more time, less VRAM. RESULTS.md
already saw this (257 ms vs 232 ms bf16).

`Float8DynamicActivationFloat8WeightConfig` quantizes activations to fp8 *at
runtime* and runs **true fp8 matmul** on Blackwell's fp8 tensor cores
(sm_120). The per-matmul work is genuinely cheaper, the quantize cost is
amortized over the matmul, and the cumulative win compounds across the
transformer's many linear layers.

The polyfill bug from RESULTS.md (`InternalTorchDynamoError: Polyfill
cmp_eq` with torchao 0.17 / torch 2.11 / cu130) is **no longer present**
under torch 2.11+cu128 / torchao 0.15.0+cu128. The Blackwell fp8 path lit up
quietly between those versions.

### Implications for production

`inference_server.py` should switch to fp8_dynamic_act for transformer
quantization. The change is two lines on top of the existing winning recipe:

```python
from torchao.quantization import quantize_, Float8DynamicActivationFloat8WeightConfig
from torchao.quantization.granularity import PerRow

# After loading the pipe, BEFORE torch.compile:
def filter_fn(module, fqn):
    if not isinstance(module, torch.nn.Linear): return False
    bad = ("pe_embedder", "norm_", "_norm", "embed", "out_proj")
    return not any(b in fqn.lower() for b in bad)

quantize_(pipe.transformer, Float8DynamicActivationFloat8WeightConfig(granularity=PerRow()), filter_fn=filter_fn)
pipe.transformer = torch.compile(pipe.transformer, mode="default", fullgraph=False, dynamic=False)
# (vae encoder/decoder compile lines unchanged — DO NOT quantize VAE)
```

Setup cost: ~80-160 s extra at startup for quantize + compile (one-time).
Per-frame: 24-30% faster. VRAM: 3.7 GB freed. No quality knob change needed.

## Phase 2 — TeaCache: SKIPPED (after research)

**Outcome: not worth porting.** Documented for the record.

Researched both `ali-vilab/TeaCache` (upstream, has FLUX.1 patch) and
`welltop-cn/ComfyUI-TeaCache` (no FLUX patch — only CogVideoX). The mechanism:
caches the residual `Δ = transformer_output - transformer_input` across
consecutive timesteps, gated on a similarity threshold computed from the
first block's `norm1` modulation output via a FLUX.1-tuned 5th-degree polynomial.

### Why it doesn't help us

| Block | Detail |
|---|---|
| **2-step inference: 0% benefit** | TeaCache hard-codes "always recompute on `cnt == 0` and `cnt == num_steps - 1`". With `num_steps=2` both steps recompute. |
| **4-step inference: theoretical max 1.5×** | Best case skip 2 of 4 calls. Probe + accumulator + polynomial overhead per step is constant; at 36 ms total each step is ~9 ms, probe overhead is a much larger fraction than at 28-step FLUX.1 (where TeaCache's claimed 2× was measured). |
| **Breaks `torch.compile`** | The patched forward has a Python `if/else` data-dependent branch (`if not should_calc`) and `.cpu().item()` on the rel_l1 scalar — both are graph breaks. Either lose compile gains, or recompile per branch flip. |
| **Polynomial is FLUX.1-tuned** | Coefficients `[4.99e+02, -2.84e+02, 5.59e+01, -3.82e+00, 2.64e-01]` were fit against FLUX.1 dev/schnell modulation distribution. Klein's distilled 4-step schedule has different modulation statistics; the polyfit gating will misfire. |
| **KV cache risk** | `Flux2KleinKVPipeline` reuses KV across steps. If TeaCache skips a transformer call, KV state for that step is never written → next step reads stale KV. Real correctness hazard. |

The fp8_dynamic_act win (24-30%) is already bigger than TeaCache's realistic
1.1-1.3× ceiling at 4-step Klein, AND fp8_dynamic_act stacks cleanly with
torch.compile. Skipping TeaCache.

## Phase 2 — TeaCache + fp8 at 4-step: SKIPPED (negative result)

After the agent's port (`workers/runpod-flux2klein/teacache_klein.py`, 659
lines, full Klein-specific KV-cache safety analysis), smoke-tested with
`Float8DynamicActivationFloat8WeightConfig` on transformer + TeaCache at
`rel_l1_thresh=0.4` + `n_steps=4`:

| recipe | 256² / 4-step | vs Phase 3b winner |
|---|---|---|
| fp8 + compile (Phase 3b winner) | **45.11 ms / 22 fps** | (baseline) |
| **fp8 + TeaCache (no transformer compile)** | **104.0 ms / 9.6 fps** | **2.3× SLOWER** |

### Why it loses

TeaCache's gating uses Python `if/else` and `.cpu().item()` on the rel_l1
scalar — both are graph breaks. To enable TeaCache, we have to skip
`torch.compile(transformer)`. At 4-step Klein:

- TeaCache theoretical max is 1.25× (skip 1 of 4 calls — the agent's KV-safety
  analysis force-recomputes cnt 0, 1, and 3, leaving only cnt=2 skippable).
- The compile loss is ≥1.4× (uncompiled transformer is 40-60% slower than
  compiled).
- 1.25× < 1.4× → compile beats caching every time, in this regime.

### The path that could in theory rescue TeaCache

Compile each `transformer_block` individually instead of the whole transformer
forward. The wrapper holding TeaCache's `if/else` stays in Python; the inner
block kernels still get fused. Untested — would require ~half a day to
implement, debug, and bench. Given the dual-GPU + xDiT phases promise
2× throughput cleanly, **TeaCache is not the right next move**. Skipping.

## Phase 4 — Standalone FORA: also skipped

Same logic as Phase 2:
- FORA's data-dependent skip breaks `torch.compile`.
- At 4 steps, FORA can fire on ≤1 step (start_step=4 by default).
- 1.25× theoretical max vs ≥1.4× compile loss → net loss.

The whole "transformer-output caching at low step counts" family of methods
(TeaCache, FORA, H2-Cache, TaylorSeer) **is not viable at Klein's distilled
2-4 step regime**. They were designed for the 28-50 step FLUX.1 / SD3 / DiT
inference where caching skips half the calls. Klein has nothing to amortize.

## ⚠️ Mid-bench reframe — 4-step is the production target

The first three bench rounds (Phase 0, Phase 3, the original Phase 5 attempt)
all targeted **2-step** inference, because that's what `RESULTS.md` called the
"winning config." That was wrong for production: live shows run **4 steps**
for image quality, per the original task spec ("steps: 4 required for
quality"). The 2-step rows above are still valid datapoints — just not the
deployment target.

Reframed plan:
- Re-bench bf16 + fp8 baselines at 4-step (this section).
- Test caching methods (TeaCache, FORA) at 4-step where they can amortize.
- Combine fp8 + caching multiplicatively.
- Move to dual-GPU.

back to roughly the production performance the user is used to. We need
**stacked** wins (fp8 + cache + dual-GPU) to push *past* current production

## Phase 7 — Dual-GPU round-robin throughput 🚀

Pod `lrlvdh3j31k14t` (2× RTX 5090, EU-RO-1, $1.98/hr, network volume
`jtgc1lxkx3` mounted at `/workspace`). Two pipelines loaded — one per GPU,
both with the Phase 3b winning recipe (fp8_dynamic_act + compile). Frames
fed alternating to GPU 0 / GPU 1 via two Python threads.

Bench: `workers/runpod-flux2klein/bench_dual_gpu.py`
Output: `/workspace/bench-2026-04-30/dual_gpu/summary.json` on the dual-GPU pod.

### 7a — Threaded round-robin (rejected — GIL bottleneck)

| device / mode | 256² mean | 256² fps | scaling |
|---|---|---|---|
| cuda:0 solo | 43.69 ms | 22.89 | — |
| cuda:1 solo | 43.78 ms | 22.84 | — |
| **threaded parallel** | **54 ms** (under contention!) | **36.86 fps** | **1.61×** |

Two Python threads share the GIL between kernel launches. Pipeline glue
(sigma scheduling, scheduler step, image post-processing) serializes, and
under contention each frame's wall time **goes up from 43 to 54 ms**.

### 7b — Multiprocess round-robin (the real win) 🏆

One process per GPU via `multiprocessing.spawn`. Each worker has its own
Python interpreter and its own `CUDA_VISIBLE_DEVICES` — no GIL between them.

| resolution | solo cuda:0 | solo cuda:1 | **parallel mp** | scaling | per-frame mean (parallel) |
|---|---|---|---|---|---|
| 256² / 4-step | 22.86 fps (43.63 ms) | 23.09 fps (43.19 ms) | **44.93 fps** | **1.97×** ✓ | 44.36 / 43.20 ms (no degradation) |
| 512² / 4-step | 8.01 fps (124.57 ms) | 8.16 fps (122.38 ms) | **15.77 fps** | **1.97×** ✓ | 126.59 / 124.73 ms (no degradation) |

**1.97× scaling at both resolutions. Per-frame latency stays at solo levels
under parallel load.** Multiprocess is the deployment architecture.

Bench: `workers/runpod-flux2klein/bench_dual_gpu_mp.py`
Output: `/workspace/bench-2026-04-30/dual_gpu_mp/summary.json` on the dual-GPU pod.

### Hardware topology check

```
GPU0    GPU1    CPU Affinity
GPU0    X       PHB             0-31
GPU1    PHB     X               0-31
```

PCIe Gen5 x8 PHB (no NVLink — consumer 5090s never have it). For round-robin
this is **fine** (no GPU↔GPU traffic; each frame is fully independent). For
xDiT/tp=2 it would be the bottleneck — every layer's all-gather goes through
the CPU PCIe complex at ~32 GB/s. This is a real reason **xDiT is not the
right next move on consumer 5090s**.

### Mapping to user production claims

User said current production is "low res ~30-40 fps" and "~512×16:9-ish ~10 fps".

| target | our number on this host (driver 570) | status |
|---|---|---|
| Low-res 30-40 fps | **45 fps @ 256² / 4-step (multiprocess)** | ✅ exceeds |
| ~512² 10 fps | **15.77 fps @ 512² / 4-step (multiprocess)** | ✅ +58% headroom |

Extrapolated to a driver-580 host (~10% faster single-GPU): **~50 fps @ 256²,
~17 fps @ 512²** as the production ceiling.

### Scaling beyond 2 GPUs

Round-robin scales linearly (subject to host CPU dispatch saturation). Per
the user's question on adding 3-4 GPUs: the multiprocess pattern keeps each
worker isolated, so adding GPUs adds throughput. Realistic estimates with
this same multiprocess design:

| GPUs | extrapolated 256² / 4-step fps | $/hr |
|---|---|---|
| 1 | 23 | $0.99 |
| 2 (measured) | 45 | $1.98 |
| 3 | ~67 | $2.97 |
| 4 | ~89 | $3.96 |
| 8 | ~150-170 (host CPU dispatch starts saturating) | $7.92 |

Past ~8 GPUs you'd want xDiT — but only if you also have NVLink (so probably
not on consumer 5090s).

## Phase 8 — xDiT distributed DiT: NOT pursued

Three independent reasons stack:

1. **Round-robin already saturates per-frame throughput** for the production
   FPS targets (Phase 7b: 1.97× scaling, exceeds both user targets).
2. **Hardware topology argues against tp**: PCIe Gen5 x8 PHB between the two
   5090s — no NVLink. Tensor parallel needs tens of all-gathers per frame; CPU
   PCIe complex at ~32 GB/s would dominate.
3. **No upstream Klein support in xDiT** — it's FLUX.1-only. Adapting to
   `Flux2KleinKVPipeline` (which has its own KV-cache attention path) is a
   multi-day port with uncertain payoff.

Defer xDiT. If/when the user wants to **lower per-frame latency** (rather
than raise throughput) — e.g. for tighter audio-reactive responsiveness —
revisit it on a hardware host with NVLink (H100, A100, or future Blackwell
data-center variants).

## Phase 9 — speculative-stack bench-off (2026-05-01)

Single dual-GPU session targeting "what else can we squeeze before AOT-Inductor
becomes the next big swing?" Three variants benched against the post-Phase-7b
production config (PerTensor + VAE fp8 + compile_mode=default → 44.93/15.77 fps
dual). All three reuse `bench_dual_gpu_mp.py` with env-flag toggles.

### Phase 9A — `compile_mode=reduce-overhead` 🏆

`BENCH_VAE_FP8=1 BENCH_COMPILE_MODE=reduce-overhead python3 bench_dual_gpu_mp.py`

| resolution | solo cuda:0 | solo cuda:1 | **parallel mp** | scaling | Δ vs prior prod (default mode) |
|---|---|---|---|---|---|
| 256² / 4-step | 25.10 fps | 25.38 fps | **50.67 fps** | 2.02× | **+13.0%** (44.93 → 50.67) |
| 512² / 4-step | 8.70 fps | 8.90 fps | **17.11 fps** | 1.97× | **+8.5%** (15.77 → 17.11) |

`reduce-overhead` folds the per-step kernel-launch sequence into a CUDA graph,
eliminating ~6 ms/frame of Python+launch overhead at 256². The PEFT view crash
that RESULTS.md saw under torch 2.10 is gone with torch 2.11 + torchao 0.17.
First-iter capture cost is invisible (rolled into the existing 80-160s warmup).
**Shipped as the new default** in `inference_server.py`.

### Phase 9B — `compile_mode=max-autotune` ❌

`BENCH_COMPILE_MODE=max-autotune python3 bench_dual_gpu_mp.py`

OOM-confirmed regression on Blackwell sm_120. The autotune search emits Triton
configs requesting **122–196 KB** of shared memory per block; the sm_120
hardware limit is **101 KB / SM** (Hopper sm_90 had ~228 KB, which is what the
default templates target). Every variant fails the autotune feasibility
check, the inductor logs spam:

```
No valid triton configs. OutOfMemoryError: out of resource: triton_mm
Required: 147456 Hardware limit:101376
Reducing block sizes or `num_stages` may help..
```

…and we fall back to a slower default kernel anyway, but with an enormous
warmup tax. Solo iters never completed within the bench's 30-min budget.
**Avoid `max-autotune` until torch ships Blackwell-aware autotune templates**
(none on the 2.11 line; expected in 2.12+). Documented in
`inference_server.py`'s `COMPILE_MODE` comment.

### Phase 9C — SageAttention v1 monkey-patch ❌

`pip install sageattention` (1.0.6) + `BENCH_USE_SAGE=1` env flag.
Wrapper monkey-patches `torch.nn.functional.scaled_dot_product_attention` →
`sageattention.sageattn(...)` for non-masked, non-dropout, non-GQA paths
**before** `torch.compile` runs.

| resolution | solo cuda:0 | solo cuda:1 | **parallel mp** | scaling | Δ vs Phase 9A baseline |
|---|---|---|---|---|---|
| 256² / 4-step | 12.74 fps | 12.72 fps | 15.96 fps | 1.25× | **-69%** (50.67 → 15.96) |
| 512² / 4-step | 6.32 fps  | 6.51 fps  | 12.31 fps | 1.95× | **-28%** (17.11 → 12.31) |

Catastrophic regression. Two stacked causes:

1. **SageAttention v1 is not torch.compile-traceable.** Its `sageattn` is a
   Python function that internally dispatches to opaque CUDA ops (per-block
   int8 quantization, then int8 attention). Inductor sees the wrapper, can't
   trace through, and emits a graph break. Every attention call (~57 in this
   transformer per inference step × 4 steps × 2 calls per step from
   self+cross attn — order of hundreds per frame) becomes its own opaque
   subgraph.
2. **`reduce-overhead` mode loses its main lever.** With graph breaks every
   few ops, CUDA graph capture can't span a meaningful chunk of work. We're
   back to Python-driven kernel launches and pay the ~6 ms/frame overhead
   that 9A was eliminating.

The 256² parallel scaling collapse from 1.97× → 1.25× confirms Python-side
serialization between GPUs — exactly the GIL-bottleneck pattern Phase 7a
hit when we tried threaded round-robin. Sage's overhead per-call fights
against multi-process gains the same way.

The Sage v1 bf16 attention itself isn't slow — but you can't get a net win
unless the kernel is `torch.library.custom_op`-registered so Inductor can
schedule around it. SageAttention v2/v3 source builds advertise improved
compile compat in their READMEs; **deferred** since the v1 result already
demonstrates the architectural problem (Python-level ops + compile = bad).
A v2 source build is ~10 min on this host and the upside is uncertain
enough not to spend pod time on it today.

### Phase 9F — SageAttention v3 (Blackwell FP4) source build ❌

The v1 wrapper (Phase 9C) regressed catastrophically. Open question: is
that a fundamental Sage-vs-compile incompatibility, or a v1-specific
issue that the Blackwell-aware v3 kernel fixes?

Built `sageattention3_blackwell` from source on the pod
(`/workspace/SageAttention/sageattention3_blackwell`). The README claims
2.7x vs FlashAttention2 on RTX5090 with FP4 microscaling attention.
sm_120a is explicitly targeted in `setup.py`. Build cost: ~2 min on the
2× 5090 host. Smoke-tested at Klein's actual head shape (24 heads,
head_dim=128) — runs cleanly.

Two configs benched, head-to-head against the no-Sage baselines:

| variant | 256² parallel | Δ vs same-mode no-Sage | 512² parallel | Δ vs same-mode no-Sage |
|---|---|---|---|---|
| Sage3 + `compile_mode=default` (F1) | 35.08 fps | **-22%** vs 44.93 fps | 17.10 fps | +8.4% vs 15.77 |
| Sage3 + `compile_mode=reduce-overhead` (F2) | 42.45 fps | **-16%** vs 50.67 fps | 17.24 fps | +0.8% vs 17.11 |

Three findings stack against shipping:

1. **Klein has mixed head dims that Sage3 can't accelerate.** During
   warmup, Sage3 spammed `Unsupported Headdim 384` and `Unsupported
   Headdim 512` per call. Klein's `single_transformer_blocks` (20 of
   them, 80% of compute) use varying head dims. Only the 5
   `transformer_blocks` (head_dim=128) get the v3 kernel; the rest fall
   back to torch's SDPA. So we're paying the wrapper overhead for an
   incomplete kernel swap.
2. **Wrapper is still Python.** Even with Sage3 (better kernel than v1),
   the SDPA wrapper that decides "kernel or fallback?" is a Python
   conditional. Inductor sees a graph break per attention call. Sage's
   own README says "supports `torch.compile` with non-cudagraphs mode"
   — but that's about the kernel itself being callable from a compiled
   graph, not about the **wrapper** that switches between Sage and
   fallback. Reduce-overhead can't capture across the wrapper boundary.
3. **256² is the most popular size.** Live VJ defaults to 512×288 (576p
   16:9), which patchifies to roughly the 512² compute bucket. Sage3
   ties there. At 256x144 (smallest preset), it loses. At ≥1024² it
   would likely win — but that's outside our latency budget anyway
   (already <8 fps at 1024² with the current stack).

**Verdict:** Sage3 helps at long sequence lengths Klein doesn't usually
hit at our target FPS. Same-class regression as v1 on the small-grid
hot path. **Not shipped.** Keep the source build artifact for re-eval
when Klein gains a uniform-head-dim variant or when Sage adds proper
custom-op compile compat.

`bench_dual_gpu_mp.py` retains `BENCH_USE_SAGE=1 BENCH_SAGE_VERSION={1,3}`
flags for future quick A/B without re-instrumenting.

### Phase 9D — AOT-Inductor 📦 (works, ops win only)

`bench_aoti.py`: capture real transformer call args via a forward-pre
hook (avoids guessing rope-id shapes and dtype), wrap in an
`ExportableTransformer` that flattens kwargs to all-positional tensor
args, `torch.export.export` → `torch._inductor.aoti_compile_and_package`
→ `aoti_load_package`. Single-GPU, fixed 256² shape, fp8 transformer
(production config), N_STEPS=4 with `kv_cache=None`.

| variant | mean (ms) | p95 (ms) | min (ms) |
|---|---|---|---|
| AOT-Inductor (.pt2) | 8.56 | 8.58 | 8.54 |
| `torch.compile(reduce-overhead)` | 8.48 | 8.53 | 8.46 |
| Δ | -1.0% (AOTI marginally slower) | within noise | within noise |

**It works.** Klein's transformer exports cleanly: 4186 nodes, 3.4 s
export, **100.7 s compile + package**, **4.0 s load**. Per-frame latency
is statistically equal to the JIT-compiled hot path — both produce
equivalent kernel pipelines.

**Caveats for shipping:**

- **Artifact size: 3.8 GB per shape** (256² alone). The 9-preset ladder
  × 1-4 steps would balloon to ~30+ GB of `.pt2` files. Either ship the
  3-4 hot presets (256x144, 512x288, 720x404, 9:16 mirrors) with
  fallback to JIT for the rest, or accept the disk cost.
- **KV cache disabled in this bench.** Klein reuses K/V across denoising
  steps; cache shape differs per step, AOT can't span shape variation.
  Either bake one .pt2 per (step, shape) pair or upstream a dynamic-cache
  AOT path. Real shipping path: bake N_STEPS .pt2's per resolution.
- **VAE encoder/decoder not benched.** Single transformer call only.
  Full-pipeline AOT would need 3 .pt2 per shape × N preset shapes.

**Verdict:** AOT-Inductor is **operationally shippable, no per-frame win**.
Defer until cold-start matters more than disk cost — e.g. for "instant
demo" deploys, ephemeral pods, or frozen production builds. Per-frame
the JIT+`reduce-overhead` stack is already at the kernel ceiling.

The `klein_transformer_256.pt2` (3.8 GB) is preserved at
`/workspace/bench-2026-04-30/aoti/` on the pod for future load-time
re-bench when needed.

### Phase 9E — fp8 KV-cache exploration

Two sub-experiments. `bench_kv_fp8.py` runs both and probes the
attention surface for the deferred third path.

**E1 — drop the `out_proj` exclusion from the fp8 filter.**

Result: **no-op**. The current filter (`bad = ("pe_embedder", "norm_",
"_norm", "embed", "out_proj")`) and a relaxed filter that drops
`"out_proj"` quantize the **same 100 Linear layers** in Klein's
transformer. The exclusion is dead string-match — Klein's attention
doesn't name its output projection `out_proj`. The Pruna-derived
exclusion list was tuned on FLUX.1 / SDXL.

Sanity-checked the prod baseline at 256²/4-step: 26.08 fps single-GPU,
38.35 ms mean (matches Bench 9A solo numbers within noise → pipeline
reproduces).

**Action:** keep the `out_proj` string in `_fp8_filter_transformer`'s
bad-list as a **defensive guard** for future model swaps. No production
change needed; documented so a future bench-runner doesn't repeat E1.

**E2 — fp8 KV-cache via attention-processor patch.**

Probed Klein's attention surface to scope the work:

- **25 attention modules** total: 5 in `transformer_blocks`, 20 in
  `single_transformer_blocks`. All class `Flux2Attention`.
- **Cache lives in `Flux2KVAttnProcessor`** (custom class), not the
  attention module itself.
- Forward signature: `(hidden_states, encoder_hidden_states=None,
  attention_mask=None, image_rotary_emb=None, **kwargs)`. The processor
  captures K/V, stores them (in bf16 today), and reuses on subsequent
  steps when `kv_cache_mode="reuse"`.
- A real fp8 KV-cache patch needs:
  1. Subclass `Flux2KVAttnProcessor` to fp8-quantize K/V on store and
     dequantize on read.
  2. Per-head scale calibration (not the per-tensor scale we use for
     transformer Linears). Attention numerics are sensitive enough that
     a single global scale degrades quality visibly at low-step counts.
  3. Composability with `torch.compile` — likely needs a custom_op
     wrapper around quantize/dequantize so Inductor doesn't graph-break.
  4. Quality regression check at α=0.05–0.18 across the resolution
     ladder + on the prompts that exercise the cache (the 4-step regime
     reuses K from step 0 in steps 1-3 in some modes).

**Estimated effort:** 1-2 days of careful work + quality A/B vs the
current bf16-cache reference. Expected upside: **15-25% of attention
cost** when the cache is hot, which is roughly **5-10% of total
inference time** at 4-step. Net: ~3-5% throughput at 256², potentially
more at 512² where attention is a larger fraction.

**Verdict:** Worth doing **after** AOT-Inductor lands and we have
per-stage profiling pinpointing attention as the next bottleneck. The
combined uncertainty (will it land cleanly? will quality hold?) is too
high to bet pod-time on today.

### What Phase 9 *didn't* try (still on the menu)

After A/B/C/D/E/F, these speculative wins remain unbenched:

- **RIFE / FILM frame interpolation** (client-side, not server bench
  scope). Frontend feature: render 2×-4× display frames between actual
  AI frames. Pure UX/perceptual smoothness. Would land in `app/vj/`.
- **`torch.compile(fullgraph=True)`** — strict no-graph-break compile.
  Today we use `fullgraph=False` because Klein's attention processor
  has Python control flow that occasionally breaks the graph. If we can
  get a fullgraph=True pass to land, reduce-overhead might capture even
  larger windows. Speculative ~3-5% upside.
- **`channels_last` memory format** for VAE encode/decode. Currently
  channels_first. On Blackwell the conv kernels prefer channels_last;
  could shave 0.5-1 ms off VAE.
- **Persistent CUDA streams across pipeline stages** — overlap VAE
  encode of frame N+1 with transformer compute on frame N. Requires
  pipeline restructure + double-buffered latents. Gets us closer to
  the round-robin ceiling at single-GPU.

None of these are obvious wins; each would need its own bench cell.

## Reframing the experiment after Pruna research

The Pruna `flux2-klein-4b-smashed` model card + tutorial revealed:

- **FORA defaults**: `fora_start_step=4`, `fora_interval=3`. At 2 inference steps
  there's literally no benefit — caching doesn't activate until step 4.
- **fp8wo + Blackwell**: Pruna validated on H100 (sm_90); fp8wo on Blackwell
  sm_120 is not Pruna-tested. RESULTS.md already saw fp8wo regress to 257 ms
  (slower than bf16) under torch 2.11 / torchao 0.17 due to dequant cost.
- **Pruna tutorial is txt2img only**. Img2img recipe is on us.
- **Pruna's claimed "2-3×"** is vs *Pruna's bf16 baseline at default 50 steps*,
  not vs our compiled 2-step. So a Pruna 4-step at "2× speedup" is competing
  with our compiled 2-step (~33 ms @ 256²), not with their own 50-step.

**The real question** the caching/quantization phases need to answer:
> Can a 4-step Klein run with cached transformer outputs match the latency
> of our compiled 2-step *and* deliver visibly better quality, so we can
> upgrade quality at constant FPS?

Phases 1-4 are reframed around that question. Quality side-by-side at fixed
prompts (lightning, fire, river, neon — the shape-compatible vocabulary from
RESULTS.md) is the deciding factor, not raw ms numbers in isolation.

# Operational gotchas — what bit us, save yourself the trip

These are NOT inference-quality findings. They're things that wasted hours
during the session and you should know about before you run into them again.

## Pod / RunPod gotchas

- **Container filesystem is ephemeral on stop/start.** `/usr/local/lib/python3.12/dist-packages`
  and `/usr/bin/node` (everything outside `/workspace`) are wiped every pod
  resume. Any pip install / apt install must re-run. `setup.sh` is idempotent
  and skips reinstalls when artifacts are detectably present, but that means
  you HAVE to run setup.sh after every pod resume — there's no "just pick up
  where I left off."
- **Driver 570 hosts can't run `cu130` torch wheels.** The error is
  `"The NVIDIA driver on your system is too old (found version 12080)"` and
  it'll trip if anything pulls in `torch==2.11.0+cu130` (which is what `pip
  install diffusers` from main does by default). Hard-pin `torch==2.11.0+cu128`
  with `--extra-index-url https://download.pytorch.org/whl/cu128` in
  `setup.sh`. RunPod doesn't expose driver-version host pinning — you get
  whatever's on the assigned host.
- **`--ignore-installed cryptography` is mandatory** on the runpod image
  because the system cryptography has no RECORD file (Debian-installed). pip
  refuses to upgrade it without that flag.
- **`pip install pruna` will silently downgrade transformers and
  huggingface_hub** to versions incompatible with diffusers main. If you
  must touch pruna, do it in a separate venv — it's not safe to mix with
  the Klein env.
- **`runpodctl pod create` returns immediately**, but SSH is not ready for
  ~60-90 s. Poll `runpodctl ssh info <id>` in a retry loop with `until echo
  "$info" | grep -q "ssh -i"` or you'll get connection-refused.
- **SSH host-key changes on every pod restart.** `ssh-keygen -R
  "[ip]:port"` to clear stale keys, or pass `-o StrictHostKeyChecking=no`.
- **Triton needs `gcc` at runtime** — not just at build time. The first
  `torch.compile` invocation calls `triton.runtime.build._build()` which
  shells out to `cc` to compile a tiny CUDA helper module (`CudaUtils`).
  Without a C compiler in `$PATH` the entire compile graph fails with:
    `torch._inductor.exc.InductorError: RuntimeError: Failed to find C
     compiler. Please specify via CC environment variable`
  …silently, after a 5-min model-load successfully completes. The stock
  `runpod/pytorch:1.0.3-cu1281-torch291-ubuntu2404` image bundles gcc,
  so this never showed up in setup.sh-based deploys. The slim
  `nvidia/cuda:12.8.1-runtime-ubuntu24.04` does NOT — must explicitly
  `apt-get install gcc g++ python3-dev`. Discovered by the first
  Dockerfile spin not getting past the warmup frame; documented in the
  `Dockerfile` apt-install comment block so future-you doesn't re-cut
  the same image.
- **No way to read container stdout from inside the container** on
  RunPod. `entrypoint.sh` `exec`s into `node`, so node's stdout is
  PID 1's stdout = docker logs = unreachable from SSH. Any error in
  the Python worker subprocess (which dies with traceback to its own
  stderr → node captures it but doesn't persist it anywhere) is invisible
  unless you SSH in and re-run `python3 /app/inference_server.py`
  manually with `2>&1 | tail -50`. Documented as the canonical debug
  step in the "Boot recipe" above.

## Network volume gotchas

- **`/workspace` (mfs#euro-2.runpod.net) is fast for big sequential files,
  pathologically slow for tiny ones.** The torch.compile Inductor cache is
  tens of thousands of small files (one per Triton kernel). Putting the
  Inductor cache directly on `/workspace` froze warmup for 10+ minutes
  before producing a single frame. Two-stage solution in setup.sh:
  - Persistent store on `/workspace/torch-inductor-cache`
  - Working copy on `/tmp/torchinductor_root` (container-disk, fast)
  - Hydrate /tmp from /workspace at boot (one big sequential copy, fast)
  - `bash setup.sh --sync-cache` pushes /tmp back to /workspace before stop
- **HF model cache** on `/workspace/hf-cache` works fine — those are big
  sequential safetensors. Don't recompute model downloads (~23 GB Klein +
  small decoder).

## torch.compile gotchas

- **`mode="reduce-overhead"`** (CUDA graphs) crashes on PEFT wrapper view
  tensors. Stay on `mode="default"`.
- **The compile cache is keyed on (op signature, input shape, dtype).** Each
  `(width, height)` you ever use pays ~150 s cold-compile cost. Use
  `WARMUP_SHAPES="W1xH1,W2xH2"` env to pre-warm the shapes you'll actually
  toggle between in production. Cache hits drop that to ~20-30 s on a warm
  Inductor cache.
- **Recompile fires on shape change, not on step count change.** sigma list
  length doesn't affect the compiled graph; you can flip 2/3/4 steps freely.

## Multi-worker gotchas (server.js)

- **Threads share the GIL** and serialize CUDA kernel dispatch — multi-GPU
  threading capped at 1.61× scaling vs the 1.97× we got with multi-process.
  One python subprocess per GPU, each pinned via `CUDA_VISIBLE_DEVICES`,
  was the only thing that recovered the missing 0.4×.
- **State-change broadcast race**: if you broadcast `{prompt, seed,
  width, height}` to all workers but worker B is still warming, B never
  sees that state and stays at defaults. Round-robin then alternates
  good/bad frames. Fix: server.js keeps a `latestState` accumulator and
  replays it to every worker on its READY transition.
- **WebRTC outbound buffer can wedge the channel** if frames pile up
  faster than the client drains. Drop on `bufferedAmount > 1 MB`. Live
  VJ wants newest-wins; stale frames are useless and clog SCTP.
- **`stdin.write` returns false at backpressure** but doesn't block — Node
  silently queues internally. Workers can be 1000+ frames behind in
  Node's internal queue while their actual stdin pipe shows EOF. Don't
  trust framePending counters as truth; observe GPU util.
- **Big binary payloads (~30 KB JSON lines) split across pipe reads** —
  accumulate `stdoutBuf` and only handle complete `\n`-terminated lines,
  otherwise JSON.parse fails on partial chunks.

## Boot / serving UX gotchas

- **The user sees "connected" the moment WebRTC handshake succeeds**, even
  if the workers are still loading weights / compiling. This was the most
  reported bug of the session ("connected but nothing happens"). Two-pronged
  fix: server emits `status="phase"` and `status="compiling"` events,
  frontend polls `/healthz` to know `readyCount/workerCount`, overlay
  shows `"preparing workers (N/M ready)"` instead of `"press generate"`
  when workers aren't there yet.
- **Compile messages emitted before the WebRTC channel is open are lost.**
  server.js now also keeps `latestCompileByWorker` so a client connecting
  mid-compile can be replayed the current state.
- **Initial compile (~150 s) emits zero Python output mid-flight.** Live
  countdown overlay drives off `est_seconds` + a local 4 Hz timer instead
  of waiting for server-side progress events.

## Resolution / display gotchas

- **1080p is unreachable from any FLUX-compatible source via integer N.**
  1080 ÷ 16 = 67.5; FLUX requires both source dims divisible by 16. So
  there's NO `(w,h)` with w%16==h%16==0 that satisfies w*N=1920, h*N=1080
  for integer N. Closest exact-N target is **2560×1440 (QHD)** which
  displays beautifully on a 1080p monitor (browser bilinear shrink).
- **Default browser canvas upscale is bilinear.** Pixelate look needs CSS
  `image-rendering: pixelated` to switch the canvas-to-display blit to
  nearest-neighbour, otherwise block edges blur.
- **Right-click → save uses canvas pixel dimensions, not displayed size.**
  Render the canvas at integer-multiple of the source (≥ 1920 long side
  by default) so saves come out at FHD or above. Integer multiple keeps
  pixelate UV-snap math aligned; non-integer introduces LINEAR blur.

```python
# Load
pipe = Flux2KleinKVPipeline.from_pretrained(KLEIN_REPO, torch_dtype=torch.bfloat16)
pipe.vae = AutoencoderKLFlux2.from_pretrained(DECODER_REPO, torch_dtype=torch.bfloat16)
pipe.to("cuda")

# Quantize (BEFORE compile, transformer linears only — skip embeds/norms)
quantize_(pipe.transformer,
          Float8DynamicActivationFloat8WeightConfig(granularity=PerRow()),
          filter_fn=_fp8_filter)

# Compile
pipe.transformer = torch.compile(pipe.transformer, mode="default")
pipe.vae.encoder = torch.compile(pipe.vae.encoder, mode="default")
pipe.vae.decoder = torch.compile(pipe.vae.decoder, mode="default")

# Per-frame: alpha-blend img2img with truncated sigma schedule
noisy = alpha * image_latents + (1 - alpha) * noise
sigmas = np.linspace(1 - alpha, 0.0, n_steps)  # n_steps=4 for production
pipe(image=None, prompt=None, prompt_embeds=cached_embeds,
     latents=noisy, sigmas=sigmas, height=h, width=w, num_inference_steps=n_steps)
```

## Serving architecture

```
WebRTC client (browser)
    │
    ▼  https://<pod>-3000.proxy.runpod.net/webrtc/offer
node server.js  (1 process)
    │  spawns one inference_server.py per GPU via CUDA_VISIBLE_DEVICES
    ├──▶ python3 inference_server.py  (CUDA_VISIBLE_DEVICES=0)  ┐
    │      stdin: JSON requests                                  │  round-robin frames,
    │      stdout: JSON status + JPEG payloads                   │  broadcast state changes
    └──▶ python3 inference_server.py  (CUDA_VISIBLE_DEVICES=1)  ┘
```

Key invariants:
- State (prompt/seed/resolution) broadcast to ALL workers; latest-state replayed on each worker's READY transition.
- Frame requests dispatched round-robin to ONE worker.
- Worker-side `request_queue` (maxsize=2) drops stale frames automatically.
- Server-side outbound drops frames when WebRTC `bufferedAmount > 1 MB`.

### How to test the image after a change

The CI workflow (`.github/workflows/runpod-flux2klein-docker.yml`) builds
+ pushes on every push to `main` that touches the worker, but the build
graph passes if pip/npm install succeed — it doesn't actually run a
forward pass. Smoke-test before promoting `:<sha>` to prod:

```bash
# 1. Spin a 1-GPU test pod, no /workspace volume (worst-case cold path):
runpodctl pod create --name vj0-test --image nerddisco/vj0-flux2klein-worker:latest \
  --gpu-id "NVIDIA GeForce RTX 5090" --gpu-count 1 \
  --container-disk-in-gb 60 \
  --ports "3000/http,22/tcp,10000/udp,10001/udp,10002/udp" \
  --data-center-ids EU-RO-1

# 2. Wait until /healthz reports inferenceReady:true (~7-12 min cold):
POD_ID=...
while ! curl -sf https://${POD_ID}-3000.proxy.runpod.net/healthz | grep -q '"inferenceReady":true'; do sleep 20; done

# 3. Smoke-test the WebRTC signaling endpoint:
curl -i -X OPTIONS https://${POD_ID}-3000.proxy.runpod.net/webrtc/offer \
  -H "Origin: https://vj0.localhost"   # expect 200 + Access-Control-Allow-Origin: *
curl -i -X POST https://${POD_ID}-3000.proxy.runpod.net/webrtc/offer \
  -H "Content-Type: application/json" -d '{}'   # expect 400 with structured error

# 4. If anything fails, SSH in and run the inference server manually to
#    surface the real Python error (entrypoint exec'd into node so its
#    stdout is the docker log, which we can't read on RunPod):
ssh root@<ip> -p <port> 'CUDA_VISIBLE_DEVICES=0 timeout 30 python3 /app/inference_server.py 2>&1 | tail -50'

# 5. Tear down:
runpodctl pod delete <POD_ID>
```

## What's NOT in the stack (and why)

| ruled out | why |
|---|---|
| Pruna runtime | Source-level transformers 5.x incompat (text2text_generation moved); ate 45 min of debug |
| TeaCache / FORA | Data-dependent skip breaks `torch.compile`; 1.25× ceiling at 4-step < ~1.4× compile loss |
| H2-Cache / TaylorSeer | Need warmup ≥ N+1 steps; can't fit in 4 steps |
| xDiT / tp=2 | PCIe Gen5 x8 PHB topology on consumer 5090s (no NVLink); all-gather per layer dominates |
| TorchAO `Float8WeightOnlyConfig` | Per-matmul dequant cost > VRAM savings on Blackwell; regressed 38% |
| TensorRT | ONNX-export fragility; days of work for ~1.5-2× ROI |
| FlashAttention 3 | Hopper-only (TMA + async warp specialization); falls back to FA2 on Blackwell |
| `bitsandbytes` | Wants `libnvJitLink.so.13` not on this image |
| SDNQ 4-bit Klein checkpoint | Custom loader outside diffusers; expected fp8wo-style regression |
